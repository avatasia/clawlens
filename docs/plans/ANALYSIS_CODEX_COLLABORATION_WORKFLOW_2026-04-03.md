## 背景

随着 `docs/` 治理、Chat Audit 修复与实施链条逐步稳定，开始出现一个新的工程组织问题：

- 是否应该为同一仓库建立两个 Codex 会话
  - 一个偏文档治理 / 审计 / 提示词
  - 一个偏代码实现 / 调试 / 测试

当前讨论尚未收敛为正式规则，因此先做待实施分析记录。

## 当前共识

### 1. 应使用同一个项目，而不是拆分仓库

当前倾向是：

- 两个 Codex 会话可同时工作
- 但都基于同一个项目目录
- 不建议为了角色分工再拆成两个仓库

当前理由：

- 文档代理需要直接看到代码现状
- 代码代理也需要直接遵守当前 `docs/` 治理规则
- 拆仓库会引入额外同步成本与漂移风险

### 2. 代码 Codex 不能不读文档

已形成的共识是：

- 代码 Codex 不能只拿一句“你负责写代码”就开工
- 至少应读取当前权威主文档与任务相关主文档

当前建议的最小文档入口包括：

- `AGENTS.md`
- `docs/architecture.md`
- 与当前任务相关的主题主文档
- 若会改 `docs/`，还应读取 `docs/GOVERNANCE_DOCS_PLAN.md`

### 3. 不应默认把全部 archive 历史都塞进上下文

当前倾向是：

- 先读主文档
- 再按需追 `history`
- 只有在当前任务确实需要追溯证据时，才读归档旧稿

### 4. 若发现文档与代码不一致，应纳入当前任务上下文

当前已形成的处理原则：

- 若当前主文档与当前代码不一致，这个冲突必须进入当前上下文
- 若只是 archive 历史稿与现状不一致，则通常只需标记为历史材料，不应误当 current authority

## 当前分歧点

### 1. 代码 Codex 的“最小上下文”到底应有多小

当前仍未达成统一：

- 一种倾向：只给少量权威入口，减少上下文噪音
- 另一种倾向：若给得过少，代码代理会丢失已知约束、历史共识和当前状态判断

目前没有最终阈值。

### 2. 是否需要为“文档 Codex / 代码 Codex”建立专门的操作说明文档

当前尚未决定：

- 是直接依赖现有 `AGENTS.md` 与 `docs/` 治理体系
- 还是应单独新增“文档代理说明 / 代码代理说明”

### 3. 是否需要固定“触发短语”

当前讨论过：

- 若用户说“按 `docs/GOVERNANCE_DOCS_PLAN.md` 执行”，则文档相关操作自动套用治理规则
- 是否还要为“双 Codex 协作模式”再定义固定触发短语，当前未决定

## 待决问题

后续若要把本主题从分析稿推进为正式规则，需要明确以下问题：

1. 是否正式支持“双 Codex 分工”作为本仓库推荐工作模式
2. 代码 Codex 的最小必读文档清单应固定为哪些文件
3. 文档 Codex 是否需要固定只能编辑 `docs/` / `README.md` / `AGENTS.md`
4. 是否需要专门的“代码代理说明文档”与“文档代理说明文档”
5. 当两个会话都可能修改同一文件时，是否需要进一步规定协调顺序
