# Engineering Lessons Playbook 2026-04-03

本文从本轮 Chat Audit 修复中提炼可复用的开发方法，不局限于 ClawLens。

适用场景：

- 有事件流、异步 hook、队列、延迟写入的功能
- 前后端同时参与、状态链较长的功能
- 宿主页面注入、扩展 UI、插件式开发
- 从研究结论进入工程实施的项目

## 建议放置位置

推荐长期保留在：

- [docs/ENGINEERING_LESSONS_PLAYBOOK_2026-04-03.md](./ENGINEERING_LESSONS_PLAYBOOK_2026-04-03.md)

不建议只埋在：

- 单次 changelog
- 某一份实现提示词
- 某一个 issue/聊天记录

原因：

- 这些经验不是单个 bug 专属，而是后续开发都会重复用到
- 单独文档更适合作为新功能分析阶段的检查清单

---

## 一、先画时序，再写代码

### 原则

只要功能涉及以下任意一项，就先画时序图，不要先写逻辑：

- 多类事件源
- hook 回调
- 异步队列
- 延迟 flush
- 运行中与完成后两套数据
- “消息先到 / run 后到”这类顺序不稳定场景

### 最低要求

分析阶段至少画出 4 条真实时序：

1. 正常成功路径
2. 数据晚到路径
3. 关键字段晚到路径
4. 非主流程事件混入路径

### 如果不这么做会怎样

典型后果：

- 逻辑只对“最理想时序”成立
- active run / pending queue / fallback 查询之间互相打架
- user 和 assistant 被拆到不同 run

---

## 二、永远做三方对账

### 原则

不要把“UI 上看到了东西”当成功。

必须同时对账：

1. 源数据
2. 落库数据
3. 最终接口/前端数据

### 推荐做法

对于消息/事件归属类功能，最少检查：

- 源文件或原始事件流里是否有这条数据
- 数据库索引表里是否写进去了
- API 返回里是否还在
- 前端 render 入参里是否还在

### 如果不这么做会怎样

常见误判：

- 以为是 CSS 问题，其实是数据没入库
- 以为是后端没返回，其实是前端状态覆盖
- 以为是排序错了，其实是最新记录根本没入索引表

---

## 三、前端问题先分层，不要一上来就调样式

### 原则

UI 异常先分成 3 类：

1. 数据层：数据没到
2. 状态层：数据到过，但被重置/覆盖
3. 渲染层：数据正确，但显示错误

### 推荐排查顺序

1. 打接口返回样本
2. 打 render 入参样本
3. 再看状态 merge / refresh / observer
4. 最后才碰样式

### 如果不这么做会怎样

会陷入：

- 一边改 CSS
- 一边改 observer
- 一边改 polling
- 但根因其实只是 compact 数据覆盖了 detail

---

## 四、宿主页面注入，优先依赖稳定骨架，不依赖易变类名

### 原则

对宿主页面做布局或 UI 注入时：

- 优先依赖结构层级
- 优先改最外层布局单元
- 不要轻易碰消息流、输入框、气泡、thread 这些内层结构

### 推荐做法

先保存一份页面 HTML 快照，再找稳定骨架。

例如：

- `body > .shell.shell--chat > header.topbar + main.content.content--chat`

这类层级通常比：

- `[class*="chat-layout"]`
- `[class*="chat-container"]`

更稳。

### 如果不这么做会怎样

典型后果：

- 右对齐被打坏
- 顶栏让位失败
- 滚动上下文异常
- 一次修布局，顺手破坏了输入框和消息气泡

---

## 五、研究结论进入实施前，必须补工程约束

### 原则

“理论上可行”不等于“可上线”。

任何研究结论进入实施前，至少补齐：

- 运行时版本要求
- 配置 schema
- 数据规模上限
- 幂等策略
- 失败回退/跳过策略

### 典型清单

1. 依赖 Node/Runtime 最低版本是什么
2. 新配置项是否写进 schema
3. 文件/日志是否可能达到 100MB 级别
4. 重复执行是否会重复写脏数据
5. 失败后是阻断启动还是记录警告

### 如果不这么做会怎样

常见后果：

- 本地能跑，线上配置过不去
- 小样本可解析，大文件直接拖垮进程
- 自动导入重复执行，结果不可控

---

## 六、调试能力要产品化

### 原则

高风险链路不能每次出问题才临时加日志。

应预置：

- 默认关闭的 debug 开关
- 关键链路的结构化日志点
- 不需要改代码即可开启的排障入口

### 推荐做法

对这些链路优先预留 debug：

- 事件接收
- 归属判定
- 写库结果
- 接口返回摘要
- 前端 render 入参

### 如果不这么做会怎样

会反复出现：

- 为了定位 bug 临时加日志
- 发版一次只看一个点
- 线上验证成本越来越高

---

## 七、把“建议路径”和“当前已实现能力”分开写

### 原则

设计文档和实施文档里，必须清楚区分：

- 当前代码已经具备什么
- 目标状态希望具备什么
- 哪些能力只是建议，不是现状

### 推荐写法

每个阶段都明确写：

- `当前现状`
- `当前缺口`
- `本阶段新增`
- `完成判据`

### 如果不这么做会怎样

典型后果：

- 文档把未实现能力写得像已经存在
- 实施者直接调用不存在的 API/字段/模块
- 评审时难以区分“设计错”还是“实现未做”

---

## 八、完成判据必须前置，不要事后补

### 原则

没有完成判据的实施步骤，基本等于不可交付。

### 每一步至少要有

- 功能判据
- 数据判据
- 错误路径判据

### 示例

- 功能判据：
  新消息进入后，`current-message-run` 在下一轮刷新内前移到新 `messageId`
- 数据判据：
  最新 run 同时存在 `user` 与 `assistant` turns
- 错误路径判据：
  logger 目录不可读时，接口返回明确错误，不阻断插件启动

---

## 九、优先修“链路正确性”，再修“产品手感”

### 原则

先保证：

- 归属正确
- 数据不丢
- 接口正确

再追：

- 动画/交互手感
- 面板自动展开
- hover 稳定性

### 如果顺序反了

你会不断看到这种情况：

- 交互手感刚变好，数据又丢了
- 布局修好了，但 turns 还是错的

---

## 十、可直接复用的开发检查表

新功能开始前，先回答这 10 个问题：

1. 这条功能链有哪些真实事件源？
2. 正常时序、晚到时序、混入时序分别是什么？
3. 哪些关键字段会晚到、为空、降级？
4. 源数据、数据库、接口三方是否能对账？
5. 前端问题属于数据层、状态层还是渲染层？
6. 有没有稳定骨架可依赖，而不是模糊 class？
7. 配置 schema、版本要求、容量上限写清了吗？
8. 重复执行是否幂等？
9. 失败时是跳过、回退还是阻断？是否明确？
10. 不改代码能不能打开排障日志？
