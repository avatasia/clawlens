# 2026-04-01 文档覆写事故复盘

## 事件摘要

在执行 ClawLens 四份审查文档复核时，操作要求本应是“核实后保存为新的文档”。实际执行中却直接覆写了原始文档：

- `docs/ARCHITECTURE_REVIEW_v2026.3.28.md`
- `docs/HOOK_ANALYSIS_v2026.3.28.md`
- `docs/PLUGIN_GUIDELINES_REVIEW.md`
- `docs/REF_ANALYSIS_llm-api-logger.md`

这些文件在操作前没有 git 提交，因此覆写行为直接损害了用户未提交的工作副本。

## 影响

1. 原文档内容被原位替换，无法通过普通 `git checkout` 或 `git restore` 恢复。
2. 后续再用 `git diff` 查看改动时，只能看到相对当前索引或历史提交的差异，不能保证反映“覆写前的真实内容”。
3. 用户需要转向编辑器本地历史、备份文件、系统快照等非 git 手段恢复原文。

## 根因

1. 误解了“保存为新的文档”的要求，把“产出新的审查结果”错误执行成“直接重写原文件”。
2. 在已知 `docs/` 下存在未提交改动的情况下，仍继续对原文件做破坏性修改。
3. 编辑前没有创建本地备份副本。
4. 操作后错误地建议用 git 方式核对和恢复，这不适用于未提交即被覆写的文件。

## 正确做法

同类任务应改为以下流程：

1. 先保持原文档不变。
2. 若用户要求“新的文件”或“新版本”，只新建带日期或版本号的新文件。
3. 如必须在原文件上做大改，先生成同目录备份文件。
4. 若发现目标文件未提交且可能是用户唯一副本，默认禁止原位覆写。
5. 未提交文件发生误改后，不把 git 当作默认恢复路径。

## 后续约束

本仓库后续执行文档类任务时，至少遵守：

1. 用户说“保存为新的文档”时，禁止覆写原文件。
2. 对未提交文档做大改前，必须先备份。
3. 工作区存在用户未提交改动时，默认采用“新文件输出”策略。
4. 提供恢复建议时，优先本地历史、备份文件、快照，不默认建议 git 恢复。

## 当前补救状态

已新建本复盘文档，并将仓库级操作约束写入根目录 `AGENTS.md`，用于约束后续同类操作。
