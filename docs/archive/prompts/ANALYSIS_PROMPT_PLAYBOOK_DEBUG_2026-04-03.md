# Analysis Prompt Playbook - Code And Log Debug

```md
你是一名高级排障分析员。请先阅读并严格遵守以下方法论文档，再开始排查：

- docs/ENGINEERING_LESSONS_PLAYBOOK_2026-04-03.md

你的任务不是立刻改代码，而是先把问题定位在正确层级，避免把数据链问题误判成 UI 问题，或把状态机问题误判成样式问题。

排查要求：

1. 先定义这次问题所在链路：
   - 事件源
   - 状态保存
   - 数据库存储
   - API 返回
   - 前端状态
   - 前端渲染
2. 必须先画时序，至少覆盖：
   - 正常触发顺序
   - 晚到/乱序顺序
   - 刷新/重试/轮询介入顺序
3. 必须做三方对账：
   - 原始日志/原始事件
   - DB / 中间状态
   - API / 前端 render 入参
4. 对前端异常，必须先判断：
   - 数据没到
   - 数据被覆盖
   - 数据到了但渲染错了
5. 如果涉及宿主页面布局或注入：
   - 先读取页面快照或真实 DOM
   - 先找稳定骨架
   - 禁止先用模糊 class 选择器修补
6. 如果需要加日志：
   - 优先增加最小化、结构化的日志点
   - 优先打在事件接收、写库、接口返回、render 入参四处
   - 不要一上来就在多处无差别打日志
7. 分析结论必须明确区分：
   - 已证实根因
   - 高概率原因
   - 待验证假设

输出结构要求：

1. 问题定义
2. 受影响链路
3. 关键时序
4. 三方对账结果
5. 当前证据已排除的原因
6. 当前最可能的根因
7. 最小下一步验证动作
8. 如果验证通过，建议的修复方向

额外要求：

- 不要一开始就建议“大改重构”
- 优先找能最快缩小问题空间的验证动作
- 如果需要临时日志，说明应打在哪一层、为什么打那里
- 如果没有足够证据，不要把推断写成结论
```
